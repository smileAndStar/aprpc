# Logger 压测报告（2026-04-13）

## 1. 测试目标
评估当前日志实现（多生产者 + 单消费者落盘）在不同并发下的吞吐和调用开销，识别瓶颈拐点。

## 2. 测试环境与方法
- 项目路径: `/home/smile/my_file/aprpc`
- 可执行文件: `./bin/logger_perf`
- 固定参数:
  - duration: `8s`
  - 线程档位: `1, 2, 4, 8, 16, 32, 64`
  - 消息大小: `128B`、`1024B`
- 指标:
  - `logs_per_s`: 每秒日志调用次数（生产速率）
  - `avg_log_call_us`: `Logger::Log` 平均调用耗时
  - `max_log_call_us`: `Logger::Log` 最大调用耗时

执行命令（已完成）：
```bash
for size in 128 1024; do
  for th in 1 2 4 8 16 32 64; do
    ./bin/logger_perf "$th" 8 "$size"
  done
done
```

## 4. 结论
- 在 `128B` 场景下，吞吐在 `32` 线程附近达到峰值（约 `295k logs/s`），`64` 线程开始回落。
- 在 `1024B` 场景下，吞吐在 `16~32` 线程进入平台区（约 `111k logs/s`），`64` 线程回落明显。
- 两种场景都出现同一趋势:
  - 线程数继续增加时，`avg_log_call_us` 近似倍增（锁竞争加剧）。
  - `logs_per_s` 不再等比例增长，说明系统已由日志链路限制。

## 5. 瓶颈判定
按“吞吐不再增长 + 调用延迟持续上升”的标准，本次压测判定：
- `128B`: 明显瓶颈拐点约在 `32` 线程之后。
- `1024B`: 明显瓶颈拐点约在 `16` 线程之后。

## 6. 瓶颈成因分析
结合实现与数据，瓶颈主要由以下三点叠加造成：

1. 队列入队使用同一把互斥锁，生产线程并发升高后锁竞争快速加剧。
  - 对应实现：`src/include/asyncqueue.h` 中 `push()` 的 `std::lock_guard<std::mutex>`。
  - 现象证据：线程数翻倍时，`avg_log_call_us` 基本持续上升，64 线程时延迟显著增大。

2. 单消费者线程落盘，消费能力上限固定，无法随生产线程数线性扩展。
  - 对应实现：`src/logger.cpp` 构造函数中仅启动 1 个写线程。
  - 现象证据：`logs_per_s` 在高并发下进入平台区（128B 在约 32 线程，1024B 在约 16~32 线程）。

3. 每条日志都执行一次 `fopen`/`fclose`，I/O 系统调用开销高，进一步压低单线程消费上限。
  - 对应实现：`src/logger.cpp` 循环内每次 `fopen(..., "a+")` 后 `fclose()`。
  - 现象证据：消息从 128B 增大到 1024B 后，平台吞吐显著降低（约 295k/s -> 111k/s）。
