# RPC 长连接改造归档

## 1. 改进内容概述

当前调用模式为“每次 RPC 调用都新建连接、收发一次后关闭连接”。本次改进目标是升级为“连接可复用的长连接模式”，并配套完善发送队列、保活、重连和服务发现缓存。

## 2. 为什么要改

1. 连接重复建立/释放成本高：
每次都要经历 TCP 三次握手、内核资源分配、连接回收，增加平均延迟和 CPU 开销。

2. 吞吐受限：
短连接在高并发下容易出现大量 TIME_WAIT，影响端口和系统资源利用率。

3. 稳定性不足：
频繁建连会放大网络抖动影响，短时故障更容易直接暴露为业务失败。

4. 扩展能力弱：
短连接模型不利于后续支持心跳、连接池、请求多路复用等能力。

## 3. 有什么好处

1. 降低单次调用时延：减少握手和连接初始化开销。
2. 提升系统吞吐：减少连接管理成本，更多 CPU 用于业务处理。
3. 提升稳定性：连接复用 + 重连机制降低瞬时波动影响。
4. 增强可扩展性：为连接池、心跳探活、负载均衡切换打基础。
5. 降低服务治理成本：配合服务发现缓存，减少对 ZooKeeper 的高频读取。

## 4. 改进目标

1. 一条 TCP 连接可承载多次 RPC 请求/响应。
2. 客户端支持连接复用和失效重连。
3. 服务端不再“每次响应后主动关闭连接”。
4. 异步发送保证缓冲区生命周期安全。
5. 增加空闲回收和基础保活能力。

## 5. 具体做法（分阶段）

### 阶段 A：先修稳定性前置问题（必须先做）

1. 修复异步发送缓冲区生命周期：
将发送数据交由会话内部持有（例如 `shared_ptr<string>` + 发送回调保持引用），避免异步写入引用悬空内存。

2. 修复 ZooKeeper 读取字符串长度处理：
`zoo_get` 返回的是字节数组和长度，必须按长度构造 `std::string`，不能依赖 `\0` 结尾。

### 阶段 B：服务端支持长连接（最小可用）

1. 改造会话写完成逻辑：
发送响应后不主动 `shutdown` 连接。

2. 维持读循环：
保持现有“长度前缀 + 持久化接收缓冲区”的解包方式，持续接收后续请求帧。

3. 异常关闭策略：
仅在协议错误、读写错误、对端关闭时释放连接。

### 阶段 C：客户端连接复用

1. 新增连接管理器（ConnectionManager）：
按 `ip:port` 维护可复用连接对象。

2. 调用流程改造：
`CallMethod` 先尝试复用连接，不可用时再重连。

3. 线程安全：
连接池访问需加锁，防止并发下重复建连和状态错乱。

4. 单连接串行发送：
同一连接上的发送使用队列串行化，避免并发写导致帧交错。

### 阶段 D：保活与容错

1. 启用 TCP keepalive。
2. 增加应用层心跳（Ping/Pong），用于更快探测断链。
3. 增加请求超时与重连退避（指数退避 + 抖动）。
4. 连接空闲超时回收，避免僵尸连接长期占用资源。

### 阶段 E：并发能力增强（可选）

1. 协议头增加 `request_id`。
2. 客户端维护 `pending_map<request_id, callback/promise>`。
3. 支持一条连接并发多请求（多路复用）。

> 若当前业务优先稳定，可先采用“单连接单飞行请求”策略，再迭代到多路复用。

## 6. 落地改造点（代码层）

1. 服务端（RpcProvider / Session）
- 保留连接，不在每次响应后关连接。
- 建立发送队列与写中状态（例如 `write_queue_ + writing_`）。
- 所有 async_write 发送缓冲都由会话内部对象持有。

2. 客户端（RpcChannel）
- 将“服务发现 + connect”从每次调用路径中抽离。
- 引入连接池与连接状态管理。
- 失败场景补充重连与错误传播策略。

3. 服务发现（ZkClient）
- 增加本地缓存与失效刷新机制。
- 处理 ZooKeeper 读取值的长度和边界。

## 7. 验收标准

1. 功能正确性：
同一客户端连续发起 N 次 RPC，请求均成功且连接不重复创建。

2. 性能指标：
与短连接版本相比，平均延迟下降，QPS 提升（建议压测对比）。

3. 稳定性指标：
长时间运行无明显连接泄漏、无异常崩溃、错误可恢复。

4. 故障演练：
服务端重启、网络闪断、ZooKeeper 短时不可用时，客户端可自动恢复。

## 8. 风险与应对

1. 风险：连接长期占用导致资源上涨。
应对：连接池上限、空闲超时、心跳失败回收。

2. 风险：并发写导致协议帧错乱。
应对：同连接发送队列串行化。

3. 风险：重试导致业务重复执行。
应对：仅对幂等请求重试，或引入业务幂等键。

## 9. 实施建议

1. 先完成阶段 A + B（最小可用长连接）。
2. 再做阶段 C + D（工程化可用）。
3. 业务有高并发需求时再推进阶段 E（多路复用）。

---

该方案遵循“先稳定，再提效，后扩展”的原则，能够在不大幅重写框架的前提下逐步升级为生产可用的长连接 RPC 通道。
