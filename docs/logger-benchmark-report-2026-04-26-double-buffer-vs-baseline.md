# 日志系统性能对比测试报告：双缓冲架构 vs 初始互斥锁队列

## 1. 测试背景
随着系统对吞吐量和低延迟的要求提升，我们将最初的基于 `std::queue` + `fopen/fclose` 的“互斥锁+条件变量生产者消费者模型”（Base），重构成了“双缓冲 (Double Buffering) + 内存映射 (Mmap)”架构。
为了探究重构后的实际提升，我们进行了一轮压测对比。

同时，在此之前我们尝试过“无锁队列 (Lock-Free Queue)”方案，但由于在高并发下存在严重的 Cache CPU 空转和伪共享降级问题，已被证明在此场景下非最优解（相关演进历史记录已归档清除）。

## 2. 方案架构简述
- **【旧版：互斥锁队列 (Base)】**：业务线程直接竞争一把全局锁挂载节点到 `std::queue<std::string>`。后台线程单次获取元素，且每次写磁盘均产生 `write` 系统调用。
- **【新版：双缓冲+Mmap】**：利用 `active_buffer_` 与 `flush_buffer_` （预分配 `std::vector`）解耦。业务线程极短临界区 `push_back`；后台线程 `O(1)` 指针交换后，离线无锁完成所有的格式化与 Mmap 内存态追加。

## 3. 测试环境与参数
- 测试工具：`bin/logger_perf`
- Message Size（每条日志大小）：128 Bytes 与 1024 Bytes
- 线程数梯度：1, 2, 4, 8, 16, 32, 64

## 6. 测试结论与架构反思

1. **极致压榨出硬件吞吐性能**：
   在常规的并发规模（4~16 个业务线程）下，双缓冲展现出了统治级的统治力，尤其是在 8 线程 128B 日志场景下，达成了极限的 **815k logs/s**（相比原版提高了 2 倍以上）。

2. **锁并不总是慢**：
   原版使用 `mutex` 和目前使用 `mutex`，效果天壤之别。原版慢在**临界区包含字符串处理及队列内存非连续**导致 Cache Bouncing。双缓冲快在**锁内只做极其廉价的内存连续的 push_back**。

3. **Mmap对 I/O 的隔离**：
   旧版的系统调用开销极高（`fopen`与`fwrite`）。而现在大块映射+批量`memcpy`避免了多余的用户态/内核态上下文切换。
