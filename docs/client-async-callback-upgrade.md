# RPC 客户端异步回调改造归档

## 1. 改进内容概述

第二个改进方案是为客户端提供异步回调调用能力：客户端发送请求后立即返回，不再阻塞等待响应；当响应到达时，由框架自动分发并触发业务回调。

该方案与长连接改造互相配合：长连接负责连接复用与传输效率，异步回调负责并发调用与调用方体验。

## 2. 为什么要改

1. 同步阻塞调用限制吞吐：
当前调用流程中，客户端调用线程会阻塞等待响应，业务并发能力受限。

2. 长尾请求拖慢线程利用率：
个别慢请求会长期占住业务线程，影响整体响应时间。

3. 难以支撑高并发场景：
在同一连接上并发多个请求时，仅靠同步等待很难高效管理。

4. 扩展能力不足：
缺少统一的超时、取消、重试、批量回调分发机制。

## 3. 有什么好处

1. 提升调用并发度：发送后立即返回，业务线程不被网络等待阻塞。
2. 降低线程数需求：通过事件驱动和回调分发减少线程堆叠。
3. 提升响应性：慢请求不再阻塞快请求的业务路径。
4. 便于治理：统一接入超时、重试、熔断、埋点统计。
5. 更易扩展：可平滑升级为 Promise/Future、协程 await 等上层风格。

## 4. 改进目标

1. 客户端支持异步发起 RPC，不阻塞调用线程。
2. 响应到达后可按请求唯一标识准确回调。
3. 支持超时失败回调与断链批量失败回调。
4. 保持对现有同步接口的兼容。

## 5. 具体做法（分阶段）

### 阶段 A：协议与数据结构准备

1. 协议头增加 request_id：
用于将响应与请求一一对应，避免并发场景下响应错配。

2. 客户端维护 pending 表：
`pending_map<request_id, PendingCall>`，保存回调、响应对象、超时信息。

3. 连接对象增加发送队列：
同一连接用 `write_queue + writing` 串行发送，避免并发写导致帧交叉。

### 阶段 B：客户端异步发送与回调分发

1. 新增异步调用接口：
例如 `AsyncCallMethod(method, controller, request, response, done)`。

2. 异步发送流程：
生成 request_id -> 注册 pending -> 入发送队列 -> 启动 async_write。

3. 持续异步读循环：
连接上持续读取响应帧，解析 request_id 并查 pending 表触发回调。

4. 回调执行策略：
轻量回调可在 I/O 线程执行；重回调投递到业务线程池，避免阻塞 I/O。

### 阶段 C：超时、异常与恢复

1. 请求超时处理：
超时后移除 pending 并触发失败回调。

2. 断链处理：
连接异常时，批量失败该连接上所有未完成请求。

3. 重连策略：
重连后仅恢复连接，不自动重放非幂等请求。

4. 重试策略：
默认仅对幂等请求开放自动重试。

### 阶段 D：兼容与演进

1. 保留同步接口：
同步接口可在内部基于异步接口 + 条件变量或 future 封装，保持对现有业务兼容。

2. 演进上层接口：
后续可增加 future 风格、协程风格接口，降低回调嵌套复杂度。

## 6. 推荐线程模型

1. 采用“事件循环 + 异步回调 + 少量 I/O 线程”。
2. 由网络库事件循环统一等待并分发连接事件。
3. 不采用“一连接一线程”模型，避免线程资源膨胀。

## 7. 落地改造点（代码层）

1. 客户端通道
- 在 `RpcChannel` 增加异步调用入口。
- 将当前阻塞式读写路径改为异步发送/异步接收。

2. 协议头
- 在 `RpcHeader` 增加 request_id 字段。

3. 连接管理
- 连接对象中实现写队列、读循环、pending 回调分发。
- 增加连接级线程安全（mutex 或 strand）。

4. 错误处理
- 统一超时、断链、解析失败的错误码与回调行为。

## 8. 验收标准

1. 功能正确性：
并发发起多次异步调用，回调结果与请求准确匹配。

2. 性能指标：
同等负载下，业务线程阻塞时间下降，吞吐上升。

3. 稳定性指标：
网络抖动、服务端重启时，回调失败路径可预期且无内存泄漏。

4. 兼容性：
原同步调用方式可继续使用，行为不回退。

## 9. 风险与应对

1. 风险：回调在线程间执行导致并发问题。
应对：约定回调线程模型，必要时统一投递业务线程池。

2. 风险：pending 表膨胀。
应对：强制超时回收、连接断开批量清理、限流保护。

3. 风险：请求重试导致重复执行。
应对：仅幂等请求自动重试，非幂等请求交由业务决策。

## 10. 实施建议

1. 先完成协议 request_id、pending 表、异步读写主流程。
2. 再补超时、断链恢复、重连退避和回调线程治理。
3. 最后统一同步/异步双接口并完善压测和故障演练。

---

该方案可作为长连接改造后的第二阶段能力升级，实现“连接复用 + 异步并发调用”的完整客户端能力闭环。
